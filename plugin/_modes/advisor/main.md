## When to use this skill

- **Annual deck refresh** — user uploads a prior-year compensation presentation; ingest for data and narrative context, run the full consulting flow, produce a new deck with current market data
- **Quick refresh (R-lite)** — prior-year deck + explicit "just the numbers updated" with no strategic changes
- **Market benchmarking** — internal pay vs. external market across roles, levels, geographies
- **Wage scale competitiveness** — evaluate a user-provided pay scale or salary grid
- **Pay scale / band design** — design or harmonize pay structures
- **Executive comp summaries** — board-ready or leadership-facing pay analyses
- **Compensation strategy** — recommendations on pay philosophy, positioning, policy
- **Pay equity analysis** — identify and frame pay gaps with remediation options
- **Total rewards comms** — employee-facing or internal education on total comp
- **Merit & incentive matrices** — increase guidelines, bonus frameworks, LTI
- **Ad-hoc wage analysis** — quick comparisons, percentile lookups, cost modeling
- **Strategic deliberation (Council)** — multi-perspective reasoning on contested comp decisions; reasoning output, optional decision memo, or integrated input into Phase 4/5

## Output format

All deliverables are PowerPoint (.pptx). Before slide construction, read the pptx skill at `/mnt/skills/public/pptx/SKILL.md`, `$ASSET_ROOT/_modes/advisor/references/template-master.md`, and `$ASSET_ROOT/_core/primitives/brand-kit.md`. Per-file resolution per `$ASSET_ROOT/_modes/advisor/references/library-resolution.md`. Default branding from `branding/_default`; switch to `deck.brand.org_slug: <slug>` for an org-specific kit, or `deck.brand: neutral` for external audiences when explicitly configured. Every deck is built new from the regenerated template — even refreshes don't patch the prior file.

### Critical orchestration rules

- **Two-register output discipline.** Operational output and deliverable output use different registers and must not blend. **Operational** = which step ran, what got written, which probe succeeded, which assumption was checked, which file was loaded — terse or silent. Surface only steps that produce a user decision or visible content. Never narrate no-op steps ("Step N — nothing to inherit, skipping" → just skip silently and proceed). **Deliverable** = section confirmations, slide narratives, council synthesis, exec summaries, audience-facing text — full prose, narrative arc, audience calibration. The skill's playbook tone (consulting peer) belongs in deliverables; operational steps run quietly. Mixing the registers — applying deliverable register to operational steps — produces ceremony that makes a 5-minute init feel like 30 minutes. Walk the protocol; do not perform it.
- **Verify-before-assert on factual claims.** Any claim involving a present-day rate, statute, role-holder, recent settlement, peer-company datum, or capability of an external tool requires a `web_search` (or, for tools, a `tool_search`) BEFORE assertion. Training-data priors are not acceptable for any value that can change between sessions. The system prompt enforces this in the abstract; the skill enforces it at every fact-load-bearing moment (Section 5 minimum wages, Section 6 peer companies, Phase 2 market data, Phase 1 union locals, persistence backend probes). When you have not verified, the correct phrasing is "I haven't checked — let me verify"; never "the rate is X, as of last year." Inventing capability or fact categories to soften an earlier wrong claim is itself an error. As of Phase 1 this discipline is also enforced *structurally* before a deliverable is written: the `refute-claim` pre-Write gate (`$ASSET_ROOT/_core/primitives/refute-claim.md`) re-verifies the `[PROXY]`/`[ESTIMATED]`/`[statutory]`-tagged claims against fetched evidence and blocks auto-finalize on a statutory `disputed`/`unverifiable`. It complements this prose discipline — it does not replace it; untagged judgment claims still ride on the rule above.
- **Artifact philosophy.** Every track and every utility command produces a durable consumption artifact — that's the END deliverable. PPTX is the END artifact for Tracks C/D/R/R-lite (Phase 6c assembled deck). Intermediate **structural** artifacts (engagement-state.yaml + checkpoint.yaml) update at every phase boundary throughout Phases 1-7 — they're the audit trail and the resume mechanism, not chat-only. No PPTX or formal END deliverable during Phases 1-5 — consulting back-and-forth is chat. Council always produces `council-state-*.yaml` (in-engagement: alongside the deck; standalone: as the END deliverable + optional decision-memo.md). Utility command END artifacts: `/init` → engagement-config-{slug}.yaml; `/update` → updated config; `/intake` → intake-form-{cycle}.pdf + meta YAML; `/quickbench` → quickbench mini-report .md. Only `/help` is chat-only (it's a meta-query, not a deliverable).
- **Phase 6 is interactive, not one-shot.** 6a confirms the section plan; 6b walks each section with two framing options + pros/cons (no static "(recommended)" label — see `$ASSET_ROOT/_core/primitives/production-gates.md` § Recommendation discipline), builds only after the user picks, previews in chat, moves on; 6c assembles. Track D runs 6b silent — section plan still confirmed in 6a, A/B prompts skipped.
- **One engagement = one budget owner** (strong nudge, not hard gate). Each budget owner owns their own envelope and approval path; bundling creates compromise scenarios neither side wants. If the user describes a multi-budget-owner engagement at any phase, surface the conflict and recommend splitting. Accept the override but don't proceed silently.
- **Cycle-stage awareness (symmetric).** When `cycle.current_stage` and `cycle.current_week_offset` are populated, calibrate every phase. At week ≤ −12 the work is intake and discovery; at week −10 to −9 it's market analysis and options modeling; at week −7 it's the approval pitch; at week ≥ 0 the wage scale is live and new strategy work for that cohort should be declined (too-late rule). At `week_offset < target_stage_offset − 4` the request is too far ahead of the spine — surface the pre-engagement mode menu (too-early protocol) instead of silently running canonical work. Both sides of the gating are detailed in `$ASSET_ROOT/_modes/advisor/references/consulting-protocol.md` § Cycle-stage gating; the menu options map to modes in `$ASSET_ROOT/_modes/advisor/references/engagement-modes.md`. Don't blindly run Phase 4 costing in week −2 (too-late) or week −22 (too-early).
- **Workforce data must use the CSV templates.** When a user uploads `.csv` or `.xlsx` workforce data, scan column names against the disallowed-fields list in `$ASSET_ROOT/_modes/advisor/references/costing-engine.md` § Disallowed-fields scan **before** parsing rows. If any disallowed field is detected, refuse to proceed and instruct the user to start a new conversation with a re-exported file using `$ASSET_ROOT/_modes/advisor/templates/wage_data_template_step.csv` or `$ASSET_ROOT/_modes/advisor/templates/wage_data_template_merit.csv`. Hard rule, not a nudge — does not relax under user pressure. Surface the warning *before* upload happens (in `/init` output, Phase 4 entry messages, onboarding) — by the time PII reaches conversation context, exposure is done.
- Council runs single-context. Personas are sequential voice blocks within one response, not parallel subagent dispatch. Never draft the synthesis block until every persona block is complete.
- Each checkpoint blocks the next phase until the user explicitly confirms.
- Step-rate rows never receive merit/compa-ratio logic; merit-based rows never receive step-progression logic. See `$ASSET_ROOT/_modes/advisor/references/costing-engine.md` § Mixed Workforce Handling.
- Every deck is built new from scratch. No track edits the uploaded prior-year file.
- When the user states something the data contradicts: present both interpretations with implications; let the user decide. Never silently comply with a narrative the data does not support.
- **Multi-province engagements** (`benchmark.scope_provinces` has 2+ provinces): every market-data pull, costing computation, and slide cost figure is per-province first, then aggregated. Never collapse to a national average when per-province values exist. Per-province `roll_up_factor`, `payroll_burden_pct`, `voluntary_turnover_pct`, and `provincial_minimum_wages` resolve from config; map-form values fall back to the `default` key when a province is absent.
- **Provincial minimum wages** are config-driven, not skill-bundled. Sourced from `benchmark.provincial_minimum_wages` per province. Skill warns when a province's `next_review` date has passed; user updates config OR skill `web_fetch`-verifies and tags `[WEB-VERIFIED YYYY-MM-DD]`. Never guess a minimum wage.
- **Selection-log discipline.** Phase 4 (on scenario pick) and Phase 6b (on section framing pick) append entries to `selection_log` in the in-memory engagement state. Phase 7 emits the array in the engagement-state YAML. Schema in `$ASSET_ROOT/_modes/advisor/references/engagement-config-template.md` § Output: end-of-engagement state file. Future engagements with matching audience archetype + scope will read this log to surface prior-pick patterns instead of relying on a static recommendation.
- **Persistence discipline — schema backend, binary chat-download.** Schema state (engagement body, cycles, decision log, brand kits, costing) writes to the `market` MCP backend via the tool calls in `references/persistence-and-ledger.md`. **Binary artifacts (PPTX, PDF, DOCX, images, sanitized CSV) never go through any backend** — they deliver as chat-download artifacts and the operator files them into their own storage (e.g. via Drive desktop sync, which auto-uploads outside any LLM context). The skill records the intended path in metadata for provenance; bytes never round-trip through an LLM context. Close-time writes use sequenced ordering (binary delivered first as chat-download, then ledger appended via `engagement_append_decision`, engagement body LAST via `engagement_put`) so partial failures leave orphan-detectable state. On write failure, halt and offer retry — never declare close without a confirmed backend response. Full contract: `$ASSET_ROOT/_modes/advisor/references/persistence-and-ledger.md`.
- **Auto-checkpoint cadence.** After every Checkpoint A/B/C/D confirmation in Phases 1, 3, 4, 5; after every Phase 6b section approval; just before Phase 7 QA. Auto-checkpoints save silently — alert the user only on write failure. User can also invoke `/checkpoint` manually. Schema and behavior in `$ASSET_ROOT/_core/primitives/engagement-loader.md` § /checkpoint command.
